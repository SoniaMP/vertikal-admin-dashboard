# MEM — Membresías

Gestión del padrón de socios del club: alta inicial, renovación temporada a temporada, edición admin y métricas agregadas. Cada membresía vincula a una persona (`Member`) con una temporada (`Season`), captura un snapshot inmutable del precio cobrado, y orquesta el cobro vía Stripe.

## Reglas

- Una membresía pertenece a una y solo una temporada.
- El DNI normalizado es único por `Member`; las renovaciones reutilizan el mismo `Member`.
- Los snapshots de la membresía (`socioFeeSnapshot`, `licensePriceSnapshot`, `licenseLabelSnapshot`, precios de suplementos) se rellenan en la creación y son inmutables salvo cuando `paymentStatus = MANUAL`.
- La transición a `ACTIVE` es automática tras el webhook de Stripe; las demás transiciones de estado son manuales.
- **Solo administradores** dan altas manuales o transicionan membresías entre estados (los instructores no intervienen en la gestión de membresías).
- Las membresías con `paymentStatus = MANUAL` requieren `paymentNote` no vacío. Es texto libre que describe cómo se cerró el cobro (regalada, transferencia, efectivo, etc.).
- El campo `paymentNote` es admin-only: se enmascara a nivel de query y nunca sale de la BD para callers que no sean ADMIN. Aplica también a CSV exports.
- El estado `PENDING_PAYMENT` es transitorio: no se permite editar oferta ni suplementos (carrera con webhook). Se resuelve esperando webhook, cancelando, o usando la acción "Marcar como pagada" para transicionar a `MANUAL`.
- `isFederated = true` indica que el admin del club ha completado la inscripción externa del socio en la federación correspondiente (proceso manual fuera del sistema). NO indica "el socio pagó la federativa" — eso se infiere de `licensePriceSnapshot > 0`.
- La cuota de socio (`MEMBERSHIP_FEE`) se cobra siempre; la licencia federativa se añade solo si el socio elige federarse vía el club.
- En el alta manual no se envían emails de confirmación (alta presencial, sin Stripe).

## Casos de uso

<details>
<summary>

### UC-MEM-01 — Alta de socio con pago online · DONE

</summary>

- **Actor:** Visitante
- **Trigger:** Accede a `/registro` y elige "Alta nueva"
- **Precondición:** Temporada activa con catálogo de licencias poblado.
- **Refs:** PAY, NOTIF

**Flujo principal**
1. Visitante rellena datos personales.
2. Selecciona Tipo, Subtipo, Categoría y suplementos sobre el catálogo activo.
3. Acepta RGPD y revisa el resumen.
4. Sistema crea `Member` (o reutiliza por DNI) y `Membership` en `PENDING_PAYMENT` con snapshot de precio y etiqueta.
5. Sistema crea sesión de Stripe Checkout (licencia + cuota socio + suplementos) y redirige.
6. Webhook tras el pago marca la membresía `ACTIVE` + `COMPLETED` y dispara los emails (notificación al club se omite si la lista está vacía).

**Flujos alternativos**
- **5a. Visitante cancela el pago en Stripe:** ver UC-PAY-01 (en `features/03.payments.md`).
- **6a. Webhook con firma inválida:** sistema responde 400 sin tocar estado. Caso termina.

## Backlog

La junta directiva no puede reportar "ingresos por cuota" vs. "ingresos por federativas" sin estimar a ojo. Decisión: añadir `socioFeeSnapshot: Int` separado del `licensePriceSnapshot` actual, junto con el estado `MANUAL` (necesario para alta manual con catálogo y para el recovery de pagos pendientes) y el campo `paymentNote` (texto libre, admin-only, que describe cómo se cerró el cobro en cualquier transición manual). ### Edición de membresías

Tras introducir `MANUAL` y `paymentNote`, definir las reglas finas de edición. Personal data se puede tocar siempre; oferta y suplementos solo en `MANUAL` (carrera con webhook hace inviable editar `PENDING_PAYMENT`); las pagadas por Stripe son inmutables salvo en personal data. La nota es editable post-creación para corregir typos o añadir contexto, siempre admin-only.

- [ ] Personal data editable en cualquier estado (`PENDING_PAYMENT`, `ACTIVE`, `EXPIRED`, `CANCELLED`).
- [ ] Oferta + suplementos editables solo cuando `paymentStatus = MANUAL`. Al cambiar, re-snapshotea los precios correspondientes.
- [ ] `paymentNote` editable post-creación desde el detalle de membresía (admin-only).
- [ ] No exponer nunca un campo de precio libre en ninguna pantalla.

### Recovery de PENDING_PAYMENT (Marcar como pagada)

Si un visitante completa el alta pública pero el pago de Stripe falla o lo abandona, la membresía queda en `PENDING_PAYMENT`. Si después paga por otra vía (transferencia, efectivo) o el club decide asumir el coste, hoy no hay forma de cerrarla — toca borrar y rehacer perdiendo la traza original (cuándo arrancó el flujo, qué eligió el socio). Decisión: acción admin "Marcar como pagada" en filas pendientes que las transiciona a `MANUAL` + `ACTIVE`, con `paymentNote` obligatoria en el diálogo. **No se re-snapshotea** el precio: se preserva el del alta original (el catálogo pudo cambiar entre el alta y el cobro offline, pero el importe acordado es el del alta).

- [ ] Botón "Marcar como pagada" en filas con `paymentStatus = PENDING_PAYMENT` del listado admin de membresías.
- [ ] Diálogo de confirmación con campo `paymentNote` obligatorio.
- [ ] Server action transiciona: `paymentStatus = MANUAL`, `status = ACTIVE`, persiste `paymentNote`. **No re-snapshotea precios.**
- [ ] Acción admin-only.
- [ ] Tests: action, ownership admin-only, transición de estado, paymentNote obligatorio, snapshots intactos.

### Desglose de ingresos cuota vs. federativa + federaciones pendientes

Cierra el bucle de los bloques anteriores: una vez que cuota y federativa viven en snapshots separados, el dashboard puede mostrar las cifras agregadas correctas. Importante: **`isFederated` indica que el admin tramitó la federación, no que el socio pagó**, así que el dashboard tiene que distinguir "lo cobrado" (`sum(licensePriceSnapshot)` — todo el dinero de federativas que ha entrado) de "lo tramitado" (`count WHERE isFederated = true` — los socios cuya federación el admin ya ha procesado). El gap entre ambos es el trabajo administrativo pendiente, que también se surface como widget para que el admin lo pueda atacar de un vistazo.

- [ ] Actualizar `fetchSeasonMetrics()` en `src/lib/admin-queries.ts` para devolver:
  - `cuotaSocios`: `sum(socioFeeSnapshot)` — total ingresado por cuota.
  - `federativasCobradas`: `sum(licensePriceSnapshot)` — total ingresado por federativas, **independientemente de si el admin las ha tramitado**.
  - `federadosOficialmente`: `count WHERE isFederated = true` — solo los socios cuya federación el admin ha procesado.
  - `federacionesPendientes`: `count WHERE licensePriceSnapshot > 0 AND isFederated = false` — los que pagaron pero faltan por tramitar.
